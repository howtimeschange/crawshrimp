# 巴拉 AI 视频工作流页面设计审查报告

日期：2026-07-15

审查对象：

- 页面源码：`app/src/renderer/views/AiVideoWorkflow.vue`
- 视觉锚点：`docs/superpowers/specs/2026-07-15-bala-ai-video-workflow-entry-design.html`
- 固定入口 spec：`docs/superpowers/specs/2026-07-15-bala-ai-video-workflow-entry-design.md`

## 结论

AI 视频工作流页面的方向是对的：它已经从散落的原子脚本，转向业务能理解的 `找图 -> AI 改图 -> 审核 -> 生视频 -> 结果` 分步工作台；款号维度、素材库弹窗、模板库弹窗和视频任务概念也都符合巴拉 AI 视频链路。

当前最大问题不是视觉风格跑偏，而是页面仍停留在“流程原型”阶段：核心长任务缺少可信状态、错误恢复、批量防错和清晰的主次动作。运营同事会相信这是一条正确链路，但还不会完全相信它能安全跑批量生产。

## 反模式判断

页面不明显像 AI 随手生成的营销页。它继承了抓虾暗色工作台的气质，任务流、款号分组和卡片密度都贴近真实运营。

但它有明显的“原型密度”：所有模块都摆出来了，真正的优先级、任务反馈和失败恢复还没长实。后续不需要重做一套视觉，而应该沿着抓虾现有 AI 生图 / AI 测图的产品 UI 习惯，补齐生产级交互。

确定性扫描结果：

- `detect.mjs` 对 `AiVideoWorkflow.vue` 和视觉锚点 HTML 返回 `[]`。
- 没有命中内置 AI slop / 确定性视觉反模式。
- CDP 浏览器可打开页面并截图；overlay 注入尝试没有稳定可见结果，因此不把 overlay 当作审查依据。

## 当前做得好的部分

1. 工作流顺序贴近业务：找图、AI 改图、审核、生视频、结果这 5 步符合当前巴拉 AI 视频链路。
2. 款号维度正确：素材回显、AI 改图和视频任务都开始按款号组织，符合批量选款、逐款审核、逐款生成的运营习惯。
3. 素材库/模板库用弹窗承载是合理的：模特库和生意管家模板库是“需要时挑选”的辅助工具，不应该常驻挤占主界面。
4. 新增“视频任务”概念是关键推进：一个款号可多任务、任务内选图、Prompt、provider 和模板，符合后续多 provider 扩展。

## 优先问题

### P1：长任务没有可信的系统状态

这条链路里每个核心动作都可能慢、失败、部分成功或需要人工介入。`开始找图并回显`、`开始生图`、`保存审核状态`、`提交生成`、`批量预检并生成`、`重新下载` 当前多数没有禁用态、进度、错误、日志和重试范围。运营人员一旦跑 20 个款，会不知道是卡住、失败还是还在排队。

修正方向：

- 每个原子能力统一状态模型：`idle / queued / running / partial / failed / done`。
- 增加卡片级进度、款号级汇总和全局日志抽屉。
- 按钮在运行时显示 loading，并禁用二次提交。
- 失败任务保留“查看原因 / 重试本款 / 重试失败项”。

### P1：AI 改图和生视频决策密度偏高

AI 改图左侧同时出现 4 个动作、模型选择、上传入口、Prompt、队列摘要、送审；生视频工具条同时出现汇总、输出目录、新增任务、批量生成、下载、结果。新手会先读界面，而不是直接做决定；熟手会觉得按钮太多但缺少默认策略。

修正方向：

- AI 改图只突出当前动作和一个主按钮，其他动作做清晰的 segmented control。
- 生视频顶部保留一个主按钮 `新增视频任务`；批量生成在任务列表有选中任务后出现。
- 视频任务弹窗按 `款号和方式 -> 图片 -> Prompt/模板 -> 创建` 分区。
- 高级项默认折叠，常用链路给默认推荐。

### P1：弹窗和选择交互的可访问性/键盘可用性不足

AI 生图页面已经有 `role=dialog`、`aria-modal`、`aria-busy`、禁用态和 `focus-visible` 习惯；AI 视频页面的自定义弹窗大多只是 `div`，没有 dialog role、焦点锁、Escape 关闭、初始焦点、可读标题关联。图片卡片还出现双击预览，键盘和触控用户容易丢操作。

修正方向：

- 图片预览、模特库、模板库、视频任务创建统一使用可访问弹窗模式。
- 弹窗补齐 `role=dialog`、`aria-modal`、`aria-labelledby`、focus trap、Escape 关闭和背景 inert。
- 所有选中卡片加 `aria-pressed` 或 checkbox 语义。
- 去掉双击依赖，统一“点击查看 / 选择”的明确区域。

### P2：视觉层级过于平均，橙色承担太多含义

橙色同时表示当前步骤、主按钮、选中态、通过数量、模板选中、状态强调。这样第一屏有抓虾辨识度，但到了审核和生视频阶段，用户不容易区分“下一步动作”“已选状态”“警示状态”“普通重点”。

修正方向：

- 橙色只给主动作和当前步骤。
- 选中态用边框、checkbox 和深色填充组合表达。
- 通过、失败、待审使用独立状态色或图标标签。
- 计数 badge 降低饱和度。
- 每个页面只保留一个视觉主按钮。

### P2：边界状态没有产品化

真实链路会遇到无素材、重复素材、无 AI 结果、未审核也被选入视频、Seedance API key 未配置、隐私保护拦截、生意管家模板预览丢失、视频 URL 过期、下载失败。当前页面主要是好天气样例，缺少坏天气设计。

修正方向：

- 为 5 步各补一组空态和异常态。
- 错误文案给出“原因 + 下一步动作”，不要只显示失败。
- Seedance 隐私拦截要明确转为“文字描述服装、场景和动作，生成原创人物版本”的安全重试方式。
- 生意管家失败要区分上传失败、提交失败、轮询超时、结果缺视频、下载失败。

## 角色风险

### 高频运营熟手

批量任务主线有了，但效率工具不足。缺少跨款批量选择、按状态一键筛选、保存视频任务预设、常用 Prompt/模板默认值、键盘批量审核。当前页面是可用台面，但还不是一天跑几十个款的生产台。

### 第一次使用的业务同事

页面出现 `semir_video_material_prepare`、`integrations/seedanceCLI`、`ARK_API_KEY` 这类实现词。业务同事能看懂“找图 / 审核 / 生视频”，但到 Seedance 和生意管家方式选择时会犹豫：什么时候必须填 Prompt？未审核图片能不能用？不选模板会怎样？

### 键盘/辅助技术用户

弹窗缺少标准 dialog 语义，图片/模板/模特卡片缺少清晰焦点和 aria 状态，双击预览不是可访问主路径。模特库、模板库和视频任务弹窗最容易丢上下文。

### 巴拉审核同事

最关心的是“这个款哪些图能进视频、哪些还要重跑、现在卡在哪一步”。当前状态标签有雏形，但缺少款号级决策摘要，例如：`208326102205：2 张可投，1 张待审，1 张需重跑，已建 2 条视频任务`。

## 下一轮建议

建议下一轮不要继续扩展新页面，而是先把已画出的工作流做成可信生产台：

1. 先补任务状态与失败恢复：长任务 loading、进度、错误、日志、重试范围。
2. 再降噪 AI 改图和生视频：每步一个主动作，视频任务弹窗做分区式创建。
3. 同步补弹窗可访问性：对齐 AI 生图页面已有的 dialog、focus、disabled、aria 规范。
4. 最后调整状态色与按钮层级：让橙色回到“主动作/当前步骤”，其他状态有独立表达。

## 验收补充

后续页面实现不仅要满足原 spec 的功能验收，还应补充以下设计验收：

1. 任一长任务运行时，用户能看到当前阶段、款号级进度、失败项数量和下一步可做动作。
2. 任一批量危险动作必须有确认、撤销或失败恢复路径。
3. 任一弹窗必须可键盘打开、关闭、完成选择，并且焦点不会丢到背景页面。
4. 任一状态色不能只靠颜色表达；必须同时有文字或图标标签。
5. 任一实现/依赖名不能作为主界面核心文案，除非它位于日志、调试详情或模板 ID 等排障区域。
