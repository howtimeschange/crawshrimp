# AI 生图工作台图片谱系、任务标题与本地缓存设计

## 背景

当前 `AI 生图工作台` 的大图预览、任务头部和图片预览存在三类耦合问题：

1. 大图缩略图和 Prompt 修改链按同一任务的所有历史 run 累加，无法区分“基于哪张图片继续修改”。
2. 结果区头部第一行实际是任务名称，第二行是参数摘要；参数摘要直接读取左侧实时表单，导致修改下次生成配置时，历史任务头部也跟着变化。
3. 结果卡优先渲染远程原图 URL，并一次挂载多个 2048×2048 结果。远程图片通常为 5–7MB；本地实测部分图片需要约 5.5–5.8 秒才能完成请求。已有本地文件也会因为 URL 排在候选首位而被忽略。

本设计在不改变 AI 生图工作台一级入口和现有 1XM 生成能力的前提下，修正图片谱系、任务信息语义和结果预览缓存策略。

## 已确认的产品语义

### 图片父子链

大图预览只展示当前图片的祖先链和当前图片，不展示同一任务的其他 run：

- 图 1 → 图 2 → 图 3：打开图 3 时显示图 1、图 2、图 3。
- 从图 1 另开分支生成图 4：打开图 4 时只显示图 1、图 4，不显示图 2、图 3。
- Prompt 修改链与底部缩略图使用同一条父子链，二者不能出现不同作用域。

### 任务头部

- 第一行明确标记为任务名称，例如 `任务：AI 生图任务 6`。
- 已生成任务的参数摘要来自已持久化的任务/run，不读取左侧正在编辑的表单。
- 左侧模型、比例、尺寸、张数属于“下次生成配置”。修改这些字段不会改写已生成结果的显示信息。
- 当下次生成配置与已生成参数不同时，可显示轻量提示；不把草稿配置伪装成历史结果参数。

### 图片缓存

- 有有效本地缓存时优先读取本地缓存。
- 没有本地缓存时才请求远程 URL。
- 远程图片首次成功显示后，在后台写入本地缓存，并持久化 URL 到本地路径的映射。
- 再次进入任务时直接使用已持久化的本地路径。
- 本地缓存文件丢失或损坏时回退远程 URL；远程成功后重新建立缓存。

## 方案选择

### 方案 A：只做前端懒加载

为图片增加懒加载和异步解码，只加载当前视口附近的图片。

优点是改动小；缺点是再次进入仍依赖远程 URL，不能满足本地缓存优先要求。

### 方案 B：复用物化接口并持久化缓存映射（采用）

沿用 `/ai-image/jobs/{job_uid}/materialize` 的远程图片白名单校验和本地缓存目录，补充确定性缓存、URL→本地路径映射、前端本地优先和远程成功后的后台缓存。

优点是复用现有安全边界，改动集中，历史任务也能逐步补齐缓存；缺点是历史远程图片首次展示后会有一次后台缓存下载。

### 方案 C：新增流式缩略图代理

由本地后端一边代理远程响应、一边写缓存，并生成缩略图。

长期性能最好，但会引入流式代理、转码、缓存清理等额外边界，超出本次调整范围。

## 数据设计

### run 的父指针

二次修改提交时，在任务参数中携带 `edit_source`：

```json
{
  "job_uid": "当前任务 UID",
  "run_uid": "父图片所属 run UID",
  "result_key": "父图片的稳定 URL 或本地结果键"
}
```

后端在合并本次 run summary 时，将 `edit_source` 快照写入新 run。每个新结果都继承所属 run 的父指针。

前端恢复任务时先建立结果索引，再从当前图片沿 `edit_source.result_key` 向上回溯，得到唯一祖先链。若历史 run 没有父指针，无法可靠推断关系，因此只显示当前选中图，不再回退成“同任务全部历史”。

### Prompt 链

每个祖先节点使用该节点所属 run 的 Prompt。当前节点追加自己的 Prompt。Prompt 标签按当前链重新编号：

- `原图 Prompt`
- `修改 Prompt 1`
- `修改 Prompt 2`

编号反映当前分支深度，不使用任务全局 run 序号。

### 缓存映射

任务 summary 增加兼容字段 `result_cache`：

```json
{
  "result_cache": {
    "https://img.example/result.png": "/local/runtime/ai-image-cache/result-<job>-<hash>.png"
  }
}
```

缓存文件名使用任务 UID 与远程 URL 哈希生成，保证同一任务、同一 URL 重复物化时复用同一文件，避免每次打开都创建重复缓存。

缓存写入流程：

1. 校验 URL 属于当前任务结果。
2. 命中并验证已有缓存文件时直接返回。
3. 未命中时下载到临时文件，校验图片格式后原子移动到确定性路径。
4. 合并并持久化 `summary.result_cache[url] = path`。
5. 返回本地路径给前端。

## 前端数据流

### 结果预览候选顺序

每张结果的预览候选顺序调整为：

1. `summary.result_cache[url]` 指向的有效本地缓存。
2. run 自带的 `output_files[index]`。
3. 远程 `image_urls[index]`。

结果卡、任务记录缩略图、大图主图和大图底部缩略图统一使用该顺序。

### 首次远程成功后的缓存

远程图片触发 `load` 后，前端为该 URL 发起一次去重后的后台物化请求。物化成功后更新内存缓存映射，并刷新对应任务数据；同一 URL 在请求进行中或已经缓存时不重复提交。

图片元素同时增加 `loading="lazy"` 和 `decoding="async"`：结果墙和任务记录只加载视口附近图片；大图当前主图保持立即加载，底部非当前缩略图可懒加载。

### 缓存失效

本地候选加载失败时：

1. 仅把该本地路径标记为本次会话失效。
2. 立即尝试同一结果的远程 URL。
3. 远程加载成功后重新物化并覆盖持久化映射。

不会因为一个缓存文件损坏而把整张结果永久标记为不可预览。

## 组件与职责

- `app/src/renderer/views/AiImageWorkbench.vue`
  - 构建父子谱系和 Prompt 链。
  - 区分已生成参数与下次生成配置。
  - 统一本地优先候选、远程回退、懒加载和缓存调度。
- `core/ai_image_service.py`
  - 把 `edit_source` 快照写入 run summary。
  - 提供确定性、可复用、原子写入的远程结果缓存。
  - 持久化 `summary.result_cache`。
- `tests/ai-image-workbench-navigation.test.js`
  - 锁定分支谱系、任务头部和预览候选顺序。
- `tests/test_ai_image_service.py`
  - 锁定父指针持久化、缓存复用、缓存映射和损坏缓存重建。

## 错误处理

- 父指针缺失：只展示当前图片，不猜测历史关系。
- 父指针指向不存在结果：在最后一个可解析祖先处截断，并保留当前图片。
- 父指针循环：使用 visited 集合停止回溯，避免界面死循环。
- 远程下载失败：继续显示已成功加载的远程图片；记录缓存失败，但不把结果卡变成永久错误。
- 本地缓存损坏：删除或忽略损坏文件，回退远程并重建。
- 任务 summary 并发更新：更新缓存映射前重新读取任务，合并当前 `result_cache`，不覆盖生成 run。

## 兼容性

- 不新增数据库表或迁移，新增信息保存在现有 `summary_json` 中。
- 旧任务没有 `edit_source` 时采用“仅当前图”保守规则。
- 已有 `output_files` 继续可用，并优先于远程 URL。
- 现有“设为主图、设为参考、下载、二次修改”接口保持不变。

## 测试与验收

### 自动化测试

1. 图 1 → 图 2 → 图 3 的大图链为 1、2、3。
2. 图 1 → 图 4 分支的大图链为 1、4。
3. 两条链的 Prompt 互不混入。
4. 旧 run 无父指针时只展示当前图。
5. 修改左侧模型和尺寸不会改变已生成参数摘要。
6. 有缓存时本地路径排在远程 URL 前。
7. 同一 URL 重复物化复用同一缓存文件。
8. 缓存映射写入后保留现有 runs。
9. 缓存文件损坏后能够重新下载并更新映射。

### 真实页面验收

1. 打开 `AI 生图任务 6`，未滚动到的旧队列不再集中请求全部原图。
2. 首次远程图成功显示后能在运行目录看到对应缓存文件。
3. 重新进入任务时先显示本地缓存，不再等待远程 URL。
4. 修改左侧模型、比例和尺寸，任务名称及已生成参数保持不变。
5. 连续修改和从祖先图分叉修改时，大图缩略图与 Prompt 链符合已确认规则。

## 非目标

- 本次不实现缩略图转码服务和缓存清理 UI。
- 本次不迁移历史 run 的父子关系；缺失关系不做启发式猜测。
- 本次不改变 1XM 生成模型、计费、轮询和输出目录策略。
