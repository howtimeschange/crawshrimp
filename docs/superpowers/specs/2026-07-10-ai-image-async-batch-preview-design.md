# AI 生图异步批量与结果预览设计

日期：2026-07-10

状态：待用户书面复核

范围：`codex/cloud-approval-workbench` 独立 worktree 内的 AI 生图工作台

## 1. 背景与目标

当前批量生成由前端逐条执行“创建独立任务、等待生成完成、再处理下一条”。这会让批量弹窗长时间停留在“批量生成中”，并把 N 条 Prompt 拆成 N 个任务。与此同时，生成结果已有远程 URL，且能下载到本地，但预览逻辑可能被尚未读取完成的本地缓存路径阻断，造成结果卡片长期空白。

本次调整目标：

1. 批量提交成功后立即关闭弹窗，返回 AI 生图工作台。
2. N 条 Prompt 归属于当前任务，表现为当前任务下的 N 条生成记录和 N 个加载队列。
3. 前端工作台单次支持 1–20 条 Prompt，不固定为 3 条；后端批量接口保留 1–100 条的接收能力。
4. 后端把批次中的全部 Prompt 并发提交给 1XM，单批最多 100 条；1XM 自己维护队列和“同时处理 5 条”的执行限制。
5. 工作台轮询批量状态，逐条把 queued、running、completed、failed 状态更新到对应队列。
6. 结果 URL 出现后主动缓存；缓存尚未准备好或缓存读取失败时仍可直接尝试远程 URL，避免空白卡片。

## 2. 方案选择

采用后端持久化异步批量队列，不采用前端 fire-and-forget。

原因：

- 后端可以在接口返回前写入 N 条 queued 记录，刷新页面后状态仍可恢复。
- 后端不建立 5 路本地执行队列，而是尽快把整批异步任务提交给 1XM，再统一轮询各任务状态。
- 同一任务的汇总结果需要串行合并，后端能避免多个完成回调同时覆盖 `summary.runs`。
- 前端只负责提交、展示和轮询，不再承担长时间运行任务的生命周期。

## 3. 用户交互

### 3.1 批量弹窗

- 默认仍可展示 3 个 Prompt 编辑卡片，但用户可新增或删除，工作台最终提交数量为 1–20 条有效 Prompt。
- 达到 20 条时禁用顶部和卡片末尾的“新增 Prompt”入口，并显示“最多添加 20 条 Prompt”；删除后恢复新增能力。
- “任务名前缀”改为当前任务信息，不再暗示将创建多个任务。
- 底部说明改为“将在当前任务下创建 N 条生成记录”。
- 点击提交后按钮只显示短暂的“提交中”。
- 后端确认入队后立即关闭弹窗，不等待任何图片生成完成。
- 入队失败时弹窗保留并显示错误，不创建伪加载队列。

### 3.2 工作台队列

- 入队成功后，当前任务的结果区立即显示 N 个队列。
- 每条 Prompt 对应一条持久化 run 记录和一张加载卡片。
- 状态文案：排队中、生成中、已完成、生成失败。
- 完成一条就立即显示该条图片，不等待整个批次结束。
- 单条失败不终止其余记录；失败队列展示可读错误信息。
- 批次全部结束后停止轮询。

### 3.3 任务边界

- 批量生成复用当前 `ai_image_jobs.job_uid`。
- 不再创建 `任务名 · Prompt 1/2/3` 等独立任务。
- 当前任务的已有历史 runs 保留，新批量 runs 追加在同一 `summary.runs` 中。
- 任务侧栏只新增生成数量，不新增 N 个任务卡片。

## 4. 后端数据模型

继续使用 `ai_image_jobs.summary.runs`，每条批量记录在供应商请求前即持久化：

```json
{
  "run_uid": "local-preassigned-run-id",
  "batch_uid": "batch-id",
  "batch_index": 0,
  "prompt": "本条 Prompt",
  "model_key": "gpt-image-2",
  "model_key_tier": "4k",
  "size": "1536x2048",
  "ratio": "3:4",
  "status": "queued",
  "task_id": "",
  "poll_url": "",
  "image_urls": [],
  "output_files": [],
  "warning": "",
  "error": ""
}
```

状态流转：

```text
queued -> running -> completed
queued -> failed
running -> failed
```

约束：

- `run_uid` 在入队时生成，供应商完成后原位更新同一记录，不追加重复记录。
- `queued` 同时覆盖本地已持久化待提交和 1XM 已接收待处理；保存 `task_id` 后即可区分供应商已接收。
- `batch_uid + batch_index` 用于稳定排序和批次追踪。
- 旧任务没有这些字段时继续按现有历史记录展示。
- 任务顶层状态在仍有 queued/running 记录时保持 `running`；全部结束后，有成功记录则为 `completed`，全部失败才为 `failed`。

## 5. API 与并发执行

新增批量入队接口：

```http
POST /ai-image/jobs/{job_uid}/batch-run
```

请求包含：

- `prompts`：后端接受 1–100 条非空 Prompt，可带显示标题；当前工作台调用方最多提交 20 条。
- 当前批量共用的模型、比例、尺寸、质量、格式、主图、参考图和输出目录。
- `request_uid`：前端在一次点击提交时生成并在网络重试中复用，后端据此保证同一批请求只入队一次。

后端先持久化 runs，再并发发起全部 1XM 异步任务创建请求。接口只等待这些“任务创建”请求返回并保存每条 `task_id` / `poll_url`，不等待图片生成完成，随后立即响应：

```json
{
  "ok": true,
  "accepted": true,
  "job_uid": "...",
  "batch_uid": "...",
  "runs": [{ "run_uid": "...", "status": "queued", "batch_index": 0, "task_id": "...", "poll_url": "..." }],
  "job": {}
}
```

提交与轮询策略：

- 后端单个批次最多接收 100 条记录，作为供应商能力与其他调用方的服务端保护上限。
- 当前 AI 生图工作台在前端把单次提交限制为 20 条，不影响后端保留 100 条接收能力。
- 对批次内全部 Prompt 并发调用 1XM 的异步任务创建接口；不等待前一条生成完成，也不在本地限制为 5 路。
- 1XM 接收任务后自行排队并最多同时处理 5 条，这个调度不由 Crawshrimp 重复实现。
- 每条任务创建成功后立即记录各自的 `task_id` 和 `poll_url`；任务创建阶段结束后，本地只负责轮询这些已提交任务。
- 轮询各任务时按供应商状态更新对应 run，任一任务进入终态后停止轮询该任务。
- 完成回写通过同一任务的锁串行合并，避免 SQLite 读改写丢失其他 run。
- 单条任务创建或轮询失败只更新该 run，不阻断其他任务的提交与轮询。
- 重复提交由客户端 `request_uid` 与服务端生成的 `batch_uid` 关联去重；相同 `request_uid` 只返回首次创建的批次，不追加 runs。

现有单图 `/run` 接口保持兼容。

## 6. 前端轮询与恢复

- 批量入队响应中的 `job` 立即替换当前任务详情。
- 工作台每 1 秒读取当前任务详情。
- 只要存在 queued/running run 就继续轮询；全部终态后停止。
- 组件卸载时停止计时器，重新进入工作台时从持久化 runs 判断是否恢复轮询。
- queued/running run 直接转成加载卡片，不依赖前端临时数组，因此刷新页面也不会丢失队列。
- 单图生成现有加载态保持不变，避免重复插入额外的通用“正在生成”队列。

## 7. 图片预览与本地缓存

预览顺序调整为：

1. 已就绪的本地缓存 data URL。
2. 远程结果 URL。
3. 加载/失败占位。

关键规则：

- 本地缓存路径存在但 data URL 尚未读取完成时，不阻断远程 URL。
- 结果 URL 写入 run 后立即调用 materialize，不再等待远程 `<img @load>` 成功后才缓存。
- materialize 成功后写入 `summary.result_cache[url] = local_path`，刷新本地预览并自动切换。
- 远程 URL 加载失败时继续等待后端本地缓存；本地读取失败时仍保留远程重试能力。
- 同一 URL 的缓存请求去重，继续使用确定性缓存文件名。

## 8. 错误处理

- 工作台添加超过 20 条 Prompt：前端阻止继续新增并提示上限。
- 后端批量请求超过 100 条或没有有效 Prompt：接口返回 400，弹窗保持打开。
- 当前任务不存在：返回 404，不新建隐式任务。
- 缺少模型 Key：入队前失败，不写 queued runs。
- 单条 1XM 任务创建失败：该 run 标记 failed，其他 Prompt 仍继续提交；批量接口返回每条提交结果。
- 单条供应商失败：仅该 run 标记 failed，其余继续。
- 后端进程重启：已持久化 queued/running 状态仍可展示；本次实现不承诺跨进程自动续跑，残留运行态应被明确标记为失败或可重新提交，不能无限加载。
- 图片远程与本地均不可用：卡片显示预览失败和重试入口，下载能力不受影响。

## 9. 测试与验收

### 自动化测试

- 后端批量 1 条、3 条、N 条和 100 条可以入队，101 条被拒绝。
- 前端工作台可以添加并提交 1–20 条；第 20 条时新增入口禁用，删除一条后恢复。
- 接口会并发提交全部任务并保存各自的 `task_id` / `poll_url`，在供应商生成完成前返回 accepted。
- 同一 job 下产生 N 条 runs，不产生 N 个 jobs。
- N 条 Prompt 不因本地 5 路限制而串行等待；在后端上限内全部提交给 1XM，由 1XM 自己排队和控制 5 条实际处理并发。
- 并发完成乱序时，runs 仍按 `batch_index` 稳定展示且不丢记录。
- 单条失败不阻断其余 runs。
- queued/running runs 渲染为 N 个加载队列。
- 本地缓存尚未读完时 `resultPreviewSrc` 会回退到远程 URL。
- 远程 `<img>` 未成功加载时也会主动 materialize。
- 缓存成功后优先显示本地预览。

### 真实客户端验收

1. 在正确的 worktree Electron 客户端提交至少 3 条 Prompt，并确认前端最多允许添加 20 条。
2. 确认弹窗在入队成功后立即关闭。
3. 确认当前任务立即出现对应数量的加载队列。
4. 确认任务侧栏没有新增 Prompt 子任务。
5. 确认结果逐条完成、逐条显示。
6. 检查 SQLite 中同一 `job_uid` 的 `summary.runs` 数量与 Prompt 数量一致。
7. 检查 `ai-image-cache` 和 `summary.result_cache`，重新进入页面应优先显示本地缓存。

## 10. 非目标

- 不改动 1XM 模型与 Key 配置体系。
- 不改动图片父子修改链规则。
- 不迁移既有独立 Prompt 任务到同一个历史任务。
- 不在本次实现跨进程可恢复的持久化任务执行器；只保证运行状态不会在 UI 中无期限假装执行。
