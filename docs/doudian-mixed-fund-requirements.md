# 抖店商城混资监控与复盘需求拆分

更新时间：2026-06-04

## 需求来源

来自《店铺运营部-商城混资监控+复盘看板开发.docx》。

核心背景：商城混资活动较多，需要在直播店铺运营 AI 工作台中建立“商城混资看板”，集中监控三个品牌各店铺的活动报名数量与后续产出效率，为报名策略沉淀数据依据。

目标入口：`https://wojia-hub.semirapp.com/tool/douyin/goods`

## 业务分层

### 1. 数据抓取类，适合抓虾先落地

第一优先级是把抖店后台的稳定数据源抓出来，形成可导出的结构化文件或后续数据接口输入。

已验证可 API-first 的部分：

- 活动广场活动列表
  - 接口：`POST /mmc/activity/seller_activity_feed?`
  - 用途：按关键词检索“混资”相关父活动，并给已配置混资入口补充父活动、活动时间等元信息。脚本最终抓取范围不再跟随搜索结果扩散，默认使用当前 4 个商城混资入口，也支持运营切换为自定义入口清单。
  - 已验证关键词“混资”可定位以下活动组：
    - `7611436032944275738`：【混资货品补贴-长周期】商家灵活出资，平台至高5倍对补
      - 子活动含 `7631472587859837230`：【高客单商品必报】优质用户混资货补
    - `7627772015895036170`：🔥全品类爆发！推荐卡全资活动报名入口
      - 子活动含 `7610636843016552714`：🔥全品类爆发！推荐卡混资活动报名入口
  - 活动范围交互：表单默认选择“当前 4 个商城混资入口”。如需增删入口，切换为“自定义入口清单”，在多行文本框里维护入口；新增一行即新增入口，删除一行即删除入口。每行格式为 `活动ID | 活动名称 | 父活动ID | 父活动名称 | 优惠券名称 | 活动链接`，只强制活动 ID 必填，其余字段可留空并由活动广场结果补充。
- 已报名商品列表
  - 接口：`POST /mmc/apply/all_product_list?`
  - 关键请求字段：`activity_id`、`product_cond.status_type=1`、`product_cond.status_list=[8]`、`filter_condition.filter_applied=false`、`filter_condition.filter_not_applied=true`、`page`、`size`
  - 口径：页面路径必须是“已报名”主 Tab 下的“报名成功”子状态；`status_list=[8]` 可直接返回页面显示的“报名成功”商品数，例如高客单入口在 2026-06-04 探查时返回 `total=784`。
  - 关键响应字段：`data.total`、`data.product_list[].applied_product_info`
  - 用途：统计每个活动入口/子活动的报名商品数，并可输出商品明细。
- 报名成功商品导出
  - 创建接口：`POST /mmc/apply/export_applied_product_list`，关键请求字段为 `activity_id`、`status_list=[8]`、`page`、`size`。
  - 轮询接口：`GET /mmc/apply/get_async_task_list?activity_id=...&task_type_list[]=2&task_type_list[]=3&task_type_list[]=13`
  - 说明：页面“在线批量操作 -> 导出列表商品(报名成功数)”会走异步导出任务，创建接口返回 `task_id`，随后轮询 `excel_url`；它适合作为兜底或人工下载链路，脚本主路径仍优先使用 `all_product_list` 直接分页拉结构化商品数据。
- 报名状态汇总
  - 接口：`POST /mmc/record/remind_info?`
  - 用途：读取待提交、待发布、审核驳回、报名异常等全局待办数量，可作为后续看板告警数据。
- 店铺上下文
  - 接口：`POST /mmc/common/shop_rel_info?`
  - 用途：读取店铺昵称、店铺 UID 列表等上下文。

已新增抓虾适配器：

- 适配包：`doudian-ops-assistant`
- 任务：`mixed_fund_signup_monitor`
- 名称：【数据】商城混资报名监控
- 输出：
  - `报名汇总`：品牌、店铺、父活动、子活动、活动时间、报名商品数、已拉取明细数
  - `报名商品明细`：活动、商品 ID、商品名称、商家编码、店铺 ID、报名状态、报名成功时间

订单复盘已新增抓虾任务：

- 任务：`mixed_fund_order_replay`
- 名称：【数据】商城混资订单复盘
- 推荐数据源：一体化订单详情 API。该模式会在一次任务里串行完成“按活动范围抓报名商品 -> 按日期窗口分页抓订单列表 -> 抓有优惠迹象的订单详情平台优惠 -> 归因复盘”，不依赖官方订单导出和邮箱验证码。
- 订单 API 模式的可执行口径：报名商品只用于建立“平台惊喜 XX 折券”的商品级归因索引，不再按活动/报名商品筛选订单列表。订单列表只按 `数据开始日期 / 数据结束日期` 分页抓取，并按接口 `total`、空页和必要的时间窗口拆分结束。这样避免报名商品过多时产生大量低价值订单查询，也避免漏掉同一时间范围内未命中报名商品过滤但订单详情实际存在平台优惠的混资订单。官方全店 GMV、精确流量体裁/流量渠道仍需官方订单导出或后续罗盘接口补齐。
- 必需字段：
  - `平台优惠`（字段 key：`platform_discount`）：用于按平台优惠券名称匹配混资活动。
  - `流量体裁`（字段 key：`content_type`）：用于识别商品卡成交金额和占比。
  - `流量渠道`（字段 key：`compass_entrance_code`）：用于计算商品卡成交渠道 Top3。
  - `流量来源`（字段 key：`c_biz`）、`流量类型`（字段 key：`ad_mark`）：作为流量归因补充维度。
  - `商品ID`、`商品名称`、`商家编码`、`成交金额`、`订单号`：用于订单明细、成交单品 Top3 和金额汇总。
- 输出：
  - `报名汇总`：同报名监控任务，用于记录本次自动报名匹配抓到的活动报名数量。
  - `报名商品明细`：同报名监控任务，用于给订单复盘里的“平台惊喜 XX 折券”做商品级辅助归因。
  - `复盘总览`：全店引导成交金额、混资成交金额、混资订单数、商品卡成交金额/占比。
  - `活动汇总`：各混资活动成交金额、订单数、商品卡成交金额/占比。
  - `商品卡渠道Top3`：商品卡成交流量渠道 Top3。
  - `成交单品Top3`：成交商品 Top3，含商品 ID、商品名称、成交金额。
  - `混资订单明细`：订单、商品、平台优惠、匹配活动、流量体裁/渠道明细。

### 2. 报表看板类，需要后续数据接口

AI 工作台的“商城混资看板”不是单纯爬取任务，需要先形成一层数据服务/接口，再做前端报表。

看板原型拆分：

- 报名监控看板
  - 维度：活动入口、品牌、店铺
  - 指标：报名数量
  - 数据源：抓虾 `mixed_fund_signup_monitor` 输出可直接支撑第一版。
- 混资数据复盘看板
  - 筛选：数据周期（周/月/年/自定义）、品牌、店铺
  - 指标：
    - 全店引导成交金额及环比
    - 商品卡成交金额/占比及环比
    - 商品卡成交渠道 top3
    - 成交单品 top3，含商品 ID、成交金额
  - 数据源需要订单与流量渠道归因。

订单归因依据来自 DOCX 截图：

- 订单明细导出字段含 `平台优惠`，可按优惠券名称归因混资活动：
  - `平台老朋友惊喜券` -> `【高客单商品必报】优质用户混资货补`
  - `平台新人首单惊喜券`、`平台新人首单福利券`、`平台限时回归礼券` -> `【混资货品补贴-长周期】商家灵活出资，平台至高5倍对补`
  - `平台惊喜 XX 折券` -> `必报！抖音商城混资券长期报名入口【商家出资5%】` 或 `🔥全品类爆发！推荐卡混资活动报名入口`
- 订单明细导出字段含 `流量体裁`、`流量渠道`，可聚合商品卡/搜索/直播/短视频/商城推荐等复盘维度。

注意：`平台惊喜 XX 折券` 在 DOCX 中对应两个活动入口，仅凭券名无法稳定区分。当前任务提供“平台惊喜折券归属”参数，默认归到 `必报！抖音商城混资券长期报名入口【商家出资5%】`；如运营当期用于推荐卡混资，应切换为推荐卡入口。

## 【数据】商城混资订单复盘参数说明

日常建议只改三个参数：`数据开始日期`、`数据结束日期`、`平台惊喜折券归属`。订单总量不再靠“最多页数/最多详情数”截断，而是只靠时间范围收口。

| 参数 | 推荐值 | 含义 |
| --- | --- | --- |
| 打开方式 | 使用当前登录页 | 直接复用 9222 里已经登录的抖店窗口；“新开订单页”只是在同一个登录态里打开订单列表入口。 |
| 数据来源 | 一体化订单详情 API（推荐） | 不走官方导出。脚本会先按活动范围抓报名商品，再按日期范围分页抓订单列表，随后抓订单详情里的平台优惠并做归因。 |
| 订单导出 Excel/CSV | 空 | 只在“已下载官方订单导出文件”模式下使用。适合你已经手工下载了含平台优惠/流量体裁/流量渠道的订单文件。 |
| 报名监控结果（可选） | 空 | 只在文件模式下使用。上传报名监控结果后，可辅助区分“平台惊喜 XX 折券”属于哪个混资活动。API 一体化模式会自动抓报名数据，不需要上传。 |
| 自动抓报名商品辅助归因 | 开启 | 先调用活动广场和报名商品接口，把报名商品 ID/商家编码做成索引，用来区分长期混资入口和推荐卡混资入口。 |
| 输出报名监控数据 | 开启 | 把“报名汇总”和“报名商品明细”一起写进订单复盘 Excel，等于两个脚本一次跑完。 |
| 活动关键词 | 混资 | 到活动广场补充活动时间和父活动信息；最终报名抓取范围以“活动范围”为准。多个关键词可用逗号分隔。 |
| 活动范围 | 当前 4 个商城混资入口 | 默认使用当前已配置的 4 个入口。需要临时增删活动时，选择“自定义入口清单”。 |
| 自定义活动入口清单 | 保留默认清单后编辑 | 只在“活动范围=自定义入口清单”时显示。每行一条活动，删除一行即删除入口，新增一行即新增入口；格式为 `活动ID | 活动名称 | 父活动ID | 父活动名称 | 优惠券名称 | 活动链接`。 |
| 平台惊喜折券归属 | 商城长期混资券入口 | 当订单商品没有命中报名商品，且优惠券名只有“平台惊喜 XX 折券”时使用的默认归属。 |
| 数据开始日期 / 数据结束日期 | 按复盘周期填写 | API 模式会转成订单列表 `create_time_start/create_time_end`；官方导出模式也用这两个日期创建导出任务。 |
| 官方导出维度 | 商品维度 | 仅“自动创建官方导出任务”模式使用。商品维度保留商品 ID、商家编码和商品金额，更适合单品复盘。 |
| 导出轮询间隔毫秒 | 10000 | 仅自动官方导出使用。当前链路可能被邮箱验证码挡住，日常不推荐用。 |
| 同时抓取无优惠订单详情 | 关闭 | 默认只抓有优惠迹象的订单详情；开启后会对时间范围内所有订单抓详情。 |

一体化 API 模式不再暴露订单数量、最大页数、最大明细数等数量限制。订单规模只由“数据开始日期 / 数据结束日期”控制；脚本内部使用固定分页批量调用接口，并按报名商品列表、订单列表接口返回的 `total` 或空页结束条件一直翻到结束。活动抓取以“活动范围”为准，活动广场只用于补充时间和父活动信息。订单列表不再按活动或报名商品筛选。

四种数据来源的定位：

- `一体化订单详情 API（推荐）`：主方案，不依赖导出验证码，能完成平台优惠和混资订单归因；若订单详情不返回流量体裁/流量渠道，则商品卡渠道 Top3 只能等待罗盘接口或官方导出补齐。
- `已下载官方订单导出文件`：人工下载到本地后上传，字段最完整，适合补齐流量体裁/流量渠道。
- `自动创建官方导出任务`：能创建抖店官方导出任务，但下载环节经常触发邮箱验证码，所以只作为兜底。
- `订单列表 API 探查`：只看订单列表接口能给什么，不足以做完整混资归因，主要用于排查接口变化。

订单侧已验证线索：

- 订单列表接口：`GET /api/order/searchlist`
  - 可返回订单、商品、支付金额、优惠金额、商品明细等结构。
  - 首屏列表接口未直接暴露截图里的 `平台优惠`、`流量体裁`、`流量渠道` 字段；但可作为分页入口和全店成交金额分母。
- 订单详情接口：`GET /api/order/detail`
  - 已验证可通过 `order_id/shop_order_id/orderId` 获取订单详情。
  - `data.order.promotion_detail.platform_discount[]`、`data.promotion[]` 可返回平台优惠券名称和金额，例如 `平台老朋友惊喜券`、`平台惊喜9折券`。
  - 可配合订单列表接口完成非导出链路的混资订单归因：订单列表分页 -> 有优惠迹象订单进入详情 -> 按平台优惠券名匹配活动 -> 输出订单明细、活动汇总、成交单品 Top。
  - 详情接口未稳定返回 `流量体裁`、`流量渠道`；当前只保留 `流量来源/c_biz_desc`、`流量类型/b_type_desc` 作为补充维度。
- 页面存在“导出订单/导出记录”，截图中的字段来自官方订单导出或导出列配置。
  - 已验证导出历史接口：`GET /order/torder/exportHistory`，历史记录的 `query_condition.custom_export_fields` 记录自定义导出字段 key。
  - 已验证下载链路：`/order/torder/exportHistory/downloadfile`。
  - 已验证导出前置校验：`/order/torder/checkIsAllowExport`。
  - 已验证导出字段接口：`GET /order/torder/queryExportFields`。
  - 已验证导出创建接口：`POST /order/torder/export`，关键请求体为 `report_type=CUSTOM`、`report_dimension=PRODUCT_ORDER`、`file_type=csv`、`custom_export_fields=[pid, order_id, create_time, pay_time, product_id, product_name, sku_code, combo_amount, platform_discount, content_type, compass_entrance_code, c_biz, ad_mark]`。
  - 已验证导出状态接口：`GET /order/torder/queryDownloadStatus?task_id_list=...`。
  - 当前账号下载接口会触发邮箱验证码；抓虾任务已支持“创建并轮询官方导出任务 + 可直接下载时解析 CSV”，遇到验证码时提示运营到抖店导出记录完成验证/下载后再用文件模式复盘。

## 建议实施路径

1. 第一阶段：抓虾数据任务
   - 已实现“商城混资报名监控”API-first 抓取。
   - 可先给运营导出 Excel，确认品牌/店铺/活动口径。
2. 第二阶段：订单复盘数据接口
   - 已实现“商城混资订单复盘”任务，默认使用订单详情 API，不依赖官方导出和邮箱验证码即可完成平台优惠归因。
   - 仍兼容官方导出文件和官方导出自动化：可自动查询字段、创建包含平台优惠/流量体裁/流量渠道的自定义订单导出任务，并在无验证码阻断时下载 CSV 后复盘。
   - 已建立优惠券名称到混资活动的映射表。
   - 订单详情 API 模式已聚合订单金额、活动成交金额、成交商品 Top；商品卡金额、流量体裁、流量渠道需后续接入罗盘聚合接口或使用官方导出补齐。
   - 验证码下载仍需人工配合，这是抖店账号安全限制，不建议绕过；当前推荐路径不再依赖该导出下载链路。
3. 第三阶段：AI 工作台看板
   - 接入抓虾/数仓输出的数据接口。
   - 实现“报名监控”和“混资复盘”两个视图。
   - 支持周期、品牌、店铺筛选。

## 当前风险与注意事项

- 抖店活动接口较重，批量抓取应限制分页和请求节奏。
- 活动名称会变化，脚本默认使用当前 4 个混资入口 ID 作为抓取范围，也允许通过自定义入口清单增删活动，并通过活动广场补充活动时间和父活动信息。
- 订单复盘的混资归因已不强依赖官方导出字段；但商品卡成交金额、商品卡渠道 Top3 仍需官方导出字段或罗盘聚合接口补齐。
- `平台惊喜 XX 折券` 无法仅凭券名区分两个活动入口。当前订单复盘任务支持上传“报名监控结果”作为辅助映射：若订单商品 ID/商家编码命中报名商品明细，则优先归到对应混资活动；仍无法区分时，再使用运营选择的默认归属。
- 看板里“三个品牌各店铺”的店铺清单不在 DOCX 明文中，后续需要从 AI 工作台配置或运营维护表补齐。
