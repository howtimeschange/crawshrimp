# 发布版本号规则

本规则适用于抓虾的正式桌面 Release。每次升版本、打 tag、更新 Release Notes 或发布 GitHub Release 前，必须先按本规则确定版本等级，并在[桌面更新发布验收清单](desktop-update-release-checklist.md)中记录结论和依据。

## 版本格式与唯一来源

- 正式发布版本使用 `vMAJOR.MINOR.PATCH`，例如 `v2.1.0`。
- `app/package.json` 与 `app/package-lock.json` 是桌面应用版本的唯一来源，文件中的版本不带 `v`，例如 `2.1.0`。
- 正式 Git tag 必须带 `v`，并与应用版本完全一致；例如应用版本为 `2.1.1` 时，tag 必须为 `v2.1.1`。

## 选择规则

| 变更等级 | 适用范围 | 升版方式 | 示例 |
| --- | --- | --- | --- |
| 脚本级 / 适配器级 | 单个脚本修复、适配器新增或调整，以及不改变产品功能边界的维护 | 只增加第三位 `PATCH` | `v2.1.0` → `v2.1.1` |
| 功能级 | 面向用户新增或显著扩展一个独立功能，例如新增一级菜单功能 | 增加第二位 `MINOR`，第三位归零 | `v2.1.0` → `v2.2.0` |
| 架构级 | 整体产品架构或核心能力体系发生变化 | 增加第一位 `MAJOR`，第二、三位归零 | `v2.1.0` → `v3.0.0` |

以下能力族属于架构级变更的典型例子：AI 生图、AI 测图、云端审批、桌面自动更新。

## 组合变更与发布前记录

- 一个 Release 同时包含多个等级的变更时，按其中**最高等级**升版；例如同时包含适配器更新和新一级菜单时，按功能级升版。
- 升高位版本时，所有低位必须归零；不得从 `v2.1.3` 直接升为 `v2.2.3` 或从 `v2.4.7` 直接升为 `v3.4.7`。
- 版本分类是产品和架构判断，不由 CI 自动推断。发布负责人必须在验收清单中填写变更等级、选择依据和目标版本后，才能更新应用版本并创建正式 tag。

## 发布执行

使用 `ship-github-cloudflare` 处理版本号更新、README / Release Notes、`main` 与匹配 `vX.Y.Z` tag 推送、CI 跟踪及正式 Release / `desktop-latest` 回读。该 skill 负责发布动作；本文件负责本仓库的版本等级决策。
